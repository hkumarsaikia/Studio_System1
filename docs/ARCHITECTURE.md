# Detailed Architecture

This document describes the implemented storyboard-first architecture in Studio_System1.

## System Summary

Studio_System1 is a programmable YouTube Shorts engine with three coordinated layers:

1. A Python orchestration layer under `src/studio/`.
2. An SVG production pipeline backed by Inkscape.
3. A Remotion/React rendering engine under `engine/`.

The core production target is fixed:

- vertical `1080x1920`
- `30fps`
- `120` seconds total duration
- `12` segments
- `10` seconds per segment
- muted MP4 output in v1

## Canonical Data Model

### Production storyboards

`data/storyboards/` is the editable source of truth for production Shorts.

Every production storyboard contains:

- `id`
- `topic`
- `title`
- `template`
- `fps`
- `width`
- `height`
- `segmentCount`
- `segmentSeconds`
- `segments`

Every production segment contains:

- `index`
- `segmentId`
- `label`
- `narrationText`
- `onScreenText`
- `subtext`
- `visual`
- `visualDirection`
- `assetRefs`
- `cameraAction`
- `palette`
- `backgroundMode`
- `motion`
- `overlays`

Defaults:

- `segmentCount = 12`
- `segmentSeconds = 10`
- `fps = 30`
- `width = 1080`
- `height = 1920`

### Compiled production payloads

`data/videos/` contains compiled render payloads only. These are derived from storyboards and are regenerated by the build pipeline.

### Demo payloads

`data/demos/` contains showcase and demo payloads. Demo IDs use the `demo_*` namespace so they do not collide with production IDs.

### Archived legacy payloads

Legacy or misplaced payloads are kept under `data/archive/` so production validation does not treat them as active inputs.

## Python Layer

### `src/studio/config.py`

Centralized repository-relative paths, including:

- `STORYBOARDS_DIR`
- `VIDEOS_DIR`
- `DEMOS_DIR`
- `ARCHIVE_DIR`
- `SEGMENTS_OUTPUT_DIR`
- `METADATA_OUTPUT_DIR`

### `src/studio/contracts.py`

Shared production constants and allowed values:

- ID regex for production and demo IDs
- production segment/frame rules
- supported visuals
- asset library mappings
- default categories and templates

This file exists so validation, generation, and rendering use the same rules.

### `src/studio/cli.py`

Unified CLI entrypoint:

```powershell
python -m src.studio.cli --help
```

Primary commands:

- `build`
- `render`
- `thumbnail`
- `metadata`
- `validate`
- `assets build`

### Generation layer

`src/studio/generators/topic_library.py` implements the production library materialization flow:

1. read `data/raw/Topics.txt`
2. create missing storyboard skeletons for `video_001` to `video_500`
3. preserve existing storyboard files unless `--force-storyboards` is used
4. compile storyboards into `data/videos/`
5. separate demos into `data/demos/`
6. refresh manifests and asset requirement outputs

`src/studio/generators/narrative_engine.py` now generates deterministic scaffolding so builds are stable across runs.

### Metadata layer

`src/studio/generators/metadata.py` generates deterministic YouTube metadata JSON from storyboard content and writes it into `output/metadata/`.

### Validation layer

Validation is schema-backed and implemented through:

- `src/studio/utils/schema_validation.py`
- `src/studio/utils/validate.py`

Validation enforces:

- exactly `500` production storyboards
- exactly `500` compiled production payloads
- exactly `12` production segments
- exactly `300` frames per compiled segment
- supported visuals only
- known asset references only
- production/demo namespace separation
- manifest contents matching files on disk

## SVG Asset Pipeline

### Entrypoints

```powershell
python build_assets.py
python -m src.studio.cli assets build
```

### Asset stages

1. Python builders write raw SVGs to `data/assets/raw/`.
2. Inkscape CLI exports processed SVGs to `data/assets/processed/`.
3. Optional SVGO optimization runs.
4. The transpiler writes React components to `engine/src/components/generated/`.
5. `index.ts` is regenerated.

### Inkscape integration

The toolchain searches `PATH` and common Windows install locations. By default it opens each processed SVG in the Inkscape GUI after export. Use `--no-view` for headless batch runs.

### Asset IDs and storyboards

Storyboards reference stable `assetRefs`. Those references are resolved against the generated asset library and then mapped into concrete scene props during compilation.

## Remotion Layer

### Engine layout

The React engine contains:

- generated SVG wrappers
- reusable motion and effect components
- scene factories
- full video composition logic
- single-segment composition logic
- generated manifest helpers

### Root behavior

`engine/src/Root.tsx` now supports:

- dataset-aware loading for production and demo payloads
- `SegmentComposition` for one segment at a time
- `MainComposition` for full demo renders
- environment-based selection through `REMOTION_VIDEO_ID`, `REMOTION_DATASET`, and `REMOTION_SEGMENT_INDEX`

### Production render flow

`src/studio/render/render_single.py` is the production render adapter.

For production IDs it performs:

1. validate/materialize the storyboard if needed
2. render `segment_01.mp4` through `segment_12.mp4` into `output/segments/<video_id>/`
3. stitch the `12` MP4 files with FFmpeg into `output/<video_id>.mp4`

For demo IDs it renders the full composition directly.

### Local NVENC layer

`engine/remotion.config.js` points at the optional local `engine/remotion-binaries-nvenc/` override. This directory is intentionally excluded from Git because it contains large local FFmpeg binaries.

## Operational Notes

- Existing storyboard files are preserved by default.
- Use `--force-storyboards` only when you intentionally want to rebuild authoring skeletons.
- Production and demo datasets are intentionally separated; do not put showcase payloads back into `data/videos/`.
- The repository is Windows-first and documented around PowerShell usage.
- TypeScript verification is expected after engine changes: `npx tsc --noEmit` from `engine/`.

## Verified Example

The current implementation has been verified by rendering `video_002` through the production segment-first flow. The output exists at `output/video_002.mp4`, and `ffprobe` reports `120.000000` seconds.
